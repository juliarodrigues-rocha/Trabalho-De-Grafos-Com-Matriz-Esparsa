   Era importante entender bem como a matriz esparsa estava estruturada antes de escrever qualquer coisa, 
   porque minha função precisaria percorrer essa estrutura corretamente.
 
   Percebi que a matriz esparsa usa um vetor de ponteiros (voos_linha), onde cada posição representa um aeroporto de origem. 
   Cada posição aponta para uma lista encadeada de nós (NoMatriz), onde cada nó guarda a coluna (id do aeroporto destino) 
   e a distancia (que no nosso projeto é o número do voo). 
   Com isso entendido, ficou mais claro o que a função 4 precisava fazer.
 
   Criei os arquivos grafo.h e grafo.c. 
   No grafo.h, escrevi os protótipos das duas funções que ficaram sob minha responsabilidade, 
   seguindo o mesmo padrão de organização que a emely usou nos outros arquivos dela.
 
   Para a função 4 (listar_voos_de_aeroporto), a lógica que desenvolvi foi:
   receber a sigla do aeroporto de origem, usar a função buscar_id_por_sigla para obter 
   o id correspondente no vetor dinâmico, e em seguida acessar a linha desse id na matriz esparsa. 
   A partir daí, basta percorrer a lista encadeada daquela linha (cada nó encontrado é um voo cadastrado saindo daquele aeroporto.)
 
   Adicionei verificações de consistência antes de percorrer a lista:
   checar se a sigla existe no vetor (id diferente de -1), checar se o id está dentro dos limites da matriz,

   o que funcionou bem porque o id de cada aeroporto é justamente sua posição no vetor, conforme implementado pela emely em aeroporto.c.

* SESSÃO 2 — Função 5: Listagem de trajetos entre dois aeroportos
    Início: 19:50 | Término: 21:42

   Essa função foi a parte mais desafiadora da minha parte do trabalho. 
   O enunciado pedia pra listar todos os trajetos possíveis entre dois aeroportos, tanto diretos quanto com escalas. 
   Pra resolver isso, decidi usar busca em profundidade (DFS — Depth First Search).
 
   A ideia central da DFS aplicada ao problema é: a partir do aeroporto de origem, visitar cada aeroporto vizinho 
   que ainda não foi visitado naquele caminho. Se chegarmos ao destino, imprimimos o caminho percorrido. 
   Se não, continuamos descendo. Quando não há mais vizinhos para explorar, voltamos(backtrack) e tentamos outro caminho.
 
   O ponto mais importante que precisei pensar foi justamente o backtrack: ao voltar de um caminho, é preciso 
   desmarcar o aeroporto como visitado, senão ele ficaria bloqueado para os próximos caminhos. 
   Isso é o que permite que a DFS encontre todos os caminhos possíveis e não apenas um.
 
   Implementei a função em duas partes dentro do grafo.c:
   A primeira é uma função auxiliar estática chamada dfs(), que recebe a matriz, o vetor, o id do aeroporto atual,
   o id do destino, um array de visitados, um array que guarda o caminho percorrido até o momento, e o tamanho atual desse caminho. 
   Ela é estática porque é apenas um detalhe interno de implementação (quem chama de fora é sempre a função principal)
 
   A segunda é a função pública listar_trajetos(), que valida as siglas recebidas, aloca os arrays auxiliares com tamanho 
   igual ao número de aeroportos cadastrados, inicializa tudo com zero e dispara a DFS.
 
   Usei arrays locais de tamanho variável (VLA — Variable Length Array) para os vetores visitado[] e caminho[], com base em vetor->tamanho.
   Isso evita alocar memória dinamicamente para estruturas temporárias simples, e funciona bem porque o tamanho 
   só precisa existir durante a execução da função.
 
   Para imprimir o caminho quando chego ao destino, percorro o array caminho[] e imprimo cada aeroporto separado por " -> "
   o que deixa a saída fácil de ler.
 
   Fiz a verificação se origem e destino são o mesmo aeroporto, retornando uma mensagem de erro nesse caso, 
   já que não faria sentido buscar trajetos.
